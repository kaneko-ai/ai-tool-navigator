### ステップ1: 環境準備とFreeLLMAPIのデプロイ

FreeLLMAPIはDockerイメージとして提供されており、Docker Composeを利用するのが最も簡単です。

まず、作業ディレクトリを作成し、`docker-compose.yaml` ファイルを作成します。

```yaml
version: '3.8'

services:
  free-llm-api:
    image: free-llm-api/free-llm-api:latest
    container_name: free-llm-api
    ports:
      - "8000:8000" # FreeLLMAPIがリッスンするポート
    volumes:
      - ./config:/app/config # 設定ファイルをマウント
      - ./data:/app/data     # ログやキャッシュデータを保存
    environment:
      - FREE_LLM_API_CONFIG_PATH=/app/config/config.yaml
      - FREE_LLM_API_ROUTING_PATH=/app/config/routing_rules.yaml
      # APIキーは環境変数で管理することを強く推奨します
      # - OPENAI_API_KEY=sk-...
      # - ANTHROPIC_API_KEY=sk-...
      # - GOOGLE_API_KEY=AIza...
      # - NVIDIA_API_KEY=nvapi-...
      # - OPENROUTER_API_KEY=sk-...
    restart: always
```

次に、`config`ディレクトリを作成し、その中に基本的な設定ファイル `config.yaml` と `routing_rules.yaml` を用意します。


デプロイは、以下のコマンドを実行するだけです。

このコマンド実行後、FreeLLMAPIのコンテナが起動し、`http://localhost:8000` でAPIエンドポイントが利用可能になります。私の環境では、このデプロイ作業は初回設定を含めて約15分で完了しました。

### ステップ3: ルーティングとフォールバック戦略の設計

FreeLLMAPIの最も強力な機能の一つが、柔軟なルーティングとフォールバックです。これは`config/routing_rules.yaml`に記述します。

私が実際に使用しているルーティングルールの一部をご紹介します。

```yaml
# config/routing_rules.yaml
rules:
  - model: "gpt-4o"
    providers:
      - name: "openai"
        model_name: "gpt-4o" # プロバイダ側のモデル名
        priority: 1
      - name: "openrouter"
        model_name: "openai/gpt-4o" # OpenRouter経由のOpenAIモデル
        priority: 2
      - name: "google"
        model_name: "gemini-1.5-pro-latest" # gpt-4oがダメならGemini
        priority: 3
    strategy: "priority" # 優先度順に試行

  - model: "claude-3-opus"
    providers:
      - name: "anthropic"
        model_name: "claude-3-opus-20240229"
        priority: 1
      - name: "openrouter"
        model_name: "anthropic/claude-3-opus"
        priority: 2
    strategy: "priority"

  - model: "llama-3-8b-instruct"
    providers:
      - name: "nvidia"
        model_name: "llama-3-8b-instruct"
        priority: 1
        # NVIDIA NIMのレートリミットが厳しい場合、OpenRouterをフォールバックに
      - name: "openrouter"
        model_name: "meta-llama/llama-3-8b-instruct"
        priority: 2
    strategy: "priority"

  - model: "default" # 特定のモデルが指定されない場合のデフォルト
    providers:
      - name: "google"
        model_name: "gemini-1.5-flash-latest" # コスト効率の良いモデルをデフォルトに
        priority: 1
      - name: "openai"
        model_name: "gpt-3.5-turbo"
        priority: 2
    strategy: "priority"
```

この設定では、例えばアプリケーションから`gpt-4o`モデルをリクエストした場合、まずOpenAIのAPIを試行します。もしOpenAIのAPIが応答しない、またはエラーを返した場合は、OpenRouter経由の`openai/gpt-4o`モデルに自動的に切り替わります。さらにそれも失敗した場合、Googleの`gemini-1.5-pro-latest`にフォールバックするという設計です。これにより、単一プロバイダの障害によるサービス停止リスクを大幅に低減できます。

ルーティング戦略は`priority`（優先度順）の他に、`cost`（コスト効率の良い順）、`latency`（応答速度の速い順）なども指定可能です。私の副業プロジェクトでは、特にコストと安定性を重視するため、`priority`と`cost`を組み合わせた戦略を多用しています。

### ステップ4: キャッシュとレートリミットの設定

FreeLLMAPIのキャッシュ機能は、特に同じ質問が頻繁に発生するシナリオで非常に有効です。`config.yaml`でキャッシュを有効にし、TTL（Time To Live）を設定します。

```yaml
# config/config.yaml (キャッシュ設定部分)
cache:
  enabled: true
  ttl: 3600 # 1時間キャッシュを保持 (秒単位)
  max_size: 1000 # キャッシュエントリの最大数
```

この設定により、同一のプロンプトに対する応答は1時間キャッシュされ、その間はプロバイダへの実際のAPI呼び出しは発生しません。これにより、APIコストの削減と応答速度の向上という二重のメリットが得られます。

また、レートリミットを設定することで、過剰なAPIリクエストを防ぎ、プロバイダの利用規約に違反したり、予期せぬ高額請求が発生したりするリスクを低減できます。

```yaml
# config/config.yaml (レートリミット設定例)
rate_limit:
  enabled: true
  default:
    requests_per_minute: 100 # デフォルトは1分あたり100リクエスト
    tokens_per_minute: 100000 # デフォルトは1分あたり10万トークン
  rules:
    - model: "gpt-4o"
      requests_per_minute: 50 # gpt-4oは1分あたり50リクエストに制限
    - user_id: "my_special_user" # 特定のユーザーには高いレートリミットを適用
      requests_per_minute: 200
```

これらの設定により、FreeLLMAPIは単なるプロキシではなく、賢いAPIゲートウェイとして機能し始めます。
